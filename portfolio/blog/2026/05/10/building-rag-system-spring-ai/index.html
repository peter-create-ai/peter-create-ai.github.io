Why RAG Matters

Retrieval-Augmented Generation (RAG) is the backbone of most practical LLM applications. Instead of fine-tuning models (expensive, slow), RAG gives the LLM access to your data at query time.

After building RAG systems for production use, here’s what I’ve learned.

Key Design Decisions

1. Document Parsing Is Harder Than You Think

PDFs with tables, scanned documents, mixed languages — these break naive parsers. I use a combination of:

  • Unstructured.io for general parsing
  • Camelot for table extraction
  • Language detection for routing to appropriate embedding models

2. Chunking Strategy Makes or Breaks Recall

Fixed-size chunks are the default but rarely optimal. What works better:

  • Recursive character splitting with overlap
  • Semantic chunking at section boundaries
  • Small-to-big retrieval — retrieve small chunks, feed larger context to LLM

3. Hybrid Retrieval Beats Pure Vector Search

Pure vector search misses exact keyword matches. My go-to setup:

  • Dense retrieval: HNSW vector index (PGVector)
  • Sparse retrieval: BM25 (full-text search)
  • Fusion: Reciprocal Rank Fusion (RRF)
  • Rerank: Cross-Encoder (BGE-Reranker)

This combination improved Recall@3 from 78% to 93% in my benchmarks.

4. Don’t Skip the Reranker

A good reranker can recover relevant documents that the retriever missed. The cost is minimal (~100ms per query for top-20 candidates) but the quality improvement is significant.

5. Always Build an Eval Set

You can’t improve what you can’t measure. My evaluation pipeline includes:

  • 200 intent classification samples (multilingual)
  • 100 RAG QA pairs with ground-truth answers
  • Metrics: Precision, Recall, F1, MRR, Faithfulness
  • Automated regression testing on every deploy
